## Q1：网络分层结构

### （一）各层作用与协议

1. 物理层（Physical Layer）

​	**作用：**物理层负责在物理介质上传输**比特流（0 和 1）**，定义了电气特性、机械特性、接口标准和传输速率等问题。它不关心数据的含义，只负责“能不能把 0/1 送过去”。

​	**代表协议 / 标准：**IEEE 802.3（以太网物理层），RS-232，光纤传输标准

​	2. 网络层（Network Layer）

​	**作用：**网络层负责在**不同网络之间**进行通信，核心任务是**寻址和路由选择**，即决定数据包从源主机到目的主机的转发路径。

​	**代表协议：**IP（Internet Protocol），ICMP（用于差错和控制信息），RIP、OSPF、BGP（路由协议）

3. 传输层（Transport Layer）

​	**作用：**传输层负责为**端到端（进程到进程）**的通信提供服务，在网络层“不可靠”的基础上，向上层提供可靠或不可靠的数据传输。

​	**代表协议：**TCP（Transmission Control Protocol），UDP（User Datagram Protocol）

4. 应用层（Application Layer）

​	**作用：**应用层直接面向用户和应用程序，为各种网络应用提供通信服务，定义了应用进程之间交换数据的格式和规则。

​	**代表协议：**HTTP / HTTPS（Web 服务），FTP（文件传输），SMTP / POP3（电子邮件），DNS（域名解析）

### （二）分层结构的优点

1. **结构清晰，降低复杂度**
   将复杂的网络通信问题分解为多个层次，便于理解、设计和实现。
2. **层次独立，便于维护和扩展**
   各层之间通过接口交互，某一层的实现变化不会影响其他层。
3. **促进标准化和互操作性**
   不同厂商只要遵循同一层的协议标准即可实现互联互通。
4. **便于故障定位和性能优化**
   出现问题时可逐层排查，提高调试效率。

### （三）分层结构的缺点

1. **增加系统开销**
   每一层都需要进行封装和解封装，带来额外的时间和空间开销。
2. **可能降低整体性能**
   严格分层可能限制跨层优化，导致效率不如整体设计。
3. **层间功能可能重复**
   例如差错控制和流量控制在不同层中可能都有实现。

## Q2：码片计算

**码片（chip）**就是用一串 +1 / −1 的短序列，把 1 个比特“展开”成多个小单位来发送。

例如：

- A 的码片序列：`(1, 1, 1, 1)`
- B 的码片序列：`(1, -1, 1, -1)`
- C 的码片序列：`(1, 1, -1, -1)`

多个站点同时发送，链路上看到的是所有站点码片“逐位相加”的结果

若是叠加结果是：

```
(2,0,2,0 ; 0,-2,0,-2 ; 0,2,0,2)
```

求解A发送的数据。

对每一组 4 个码片：

1. 与 A 的码片 `(1,1,1,1)` **逐位相乘**
2. 相加
3. 再除以码长（4）

第一组：

```
接收： (2, 0, 2, 0)
A码： (1, 1, 1, 1)
```

计算内积：

```
2×1 + 0×1 + 2×1 + 0×1 = 4
4 / 4 = 1  → 判为比特 1
```

第二组：

```
(0, -2, 0, -2) · (1,1,1,1)
= -4
-4 / 4 = -1 → 判为比特 0
```

第三组：

```
(0, 2, 0, 2) · (1,1,1,1)
= 4
4 / 4 = 1
```

所以结果是：

```
101
```

## Q3：路由协议

RIP、OSPF 和 BGP 都属于**路由协议**，用于交换路由信息，但它们在**适用范围、算法和应用场景**等方面存在显著差异。

### 一、基本分类角度的不同

从**作用范围**来看，三者可分为：

- **RIP、OSPF**：内部网关协议（IGP），用于**自治系统（AS）内部**
- **BGP**：外部网关协议（EGP），用于**自治系统之间**

### 二、路由算法与工作机制的不同

**（一）RIP（Routing Information Protocol）**

RIP 是一种基于**距离向量算法**的内部网关协议，使用**跳数**作为路由度量标准，最大跳数为 15。路由器通过**周期性向邻居广播完整路由表**来更新路由信息，算法简单但收敛速度慢，易产生路由环路，适用于小规模网络。

- **路由算法**：距离向量算法
- **工作方式**：周期性向邻居发送整个路由表
- **更新周期**：默认 30 秒

特点是实现简单，但效率和可扩展性较差。

**（二）OSPF（Open Shortest Path First）**

OSPF 是一种基于**链路状态算法**的内部网关协议，通过泛洪链路状态信息使各路由器建立全网拓扑，并利用最短路径算法计算路由。OSPF 以**开销（Cost）**作为度量，收敛速度快，支持区域划分，适合中大型网络。

- **路由算法**：链路状态算法（Dijkstra 最短路径算法）
- **工作方式**：通过泛洪方式传播链路状态信息，构建全网拓扑图
- **更新方式**：触发更新为主，减少冗余流量

适用于中大型网络，收敛速度快，结构清晰。

**（三）BGP（Border Gateway Protocol）**

BGP 是用于**自治系统之间**的外部网关协议，采用**路径向量算法**，通过交换包含路径属性的路由信息进行路由选择。BGP 基于 TCP 连接，强调**策略控制与稳定性**，是支撑整个 Internet 路由体系的核心协议。

- **路由算法**：路径向量算法
- **工作方式**：基于 TCP 连接（端口 179），按策略交换路由信息
- **关注重点**：路径属性和策略，而非最短路径

是 Internet 的核心协议，强调**可控性与稳定性**。

### 三、应用场景对比

| 协议 | 主要应用场景                     |
| ---- | -------------------------------- |
| RIP  | 小型、简单网络，教学或实验环境   |
| OSPF | 企业网、校园网、运营商内部网络   |
| BGP  | 不同自治系统之间的 Internet 路由 |

## Q5：TCP协议

### 一、TCP 流量控制（Flow Control）

**（一）目的**

TCP 流量控制的目的是：**防止发送方发送数据过快，导致接收方来不及处理而造成数据丢失。**流量控制关注的是**发送方与接收方之间的处理能力匹配问题**，属于端到端的控制机制。

**（二）实现原理**

TCP 通过**滑动窗口机制**实现流量控制，具体做法是：

1. 接收方在 TCP 首部中通过 **接收窗口（rwnd）** 字段通告自身可用的缓冲区大小；
2. 发送方根据接收方通告的 rwnd 限制自己一次最多可发送的数据量；
3. 当接收方缓冲区变小时，rwnd 变小，发送方减慢发送速率；
4. 当接收方缓冲区释放后，rwnd 变大，发送方可继续发送数据。

因此，发送窗口大小不超过接收窗口大小，从而实现流量控制。

### 二、TCP 拥塞控制（Congestion Control）

**（一）目的**

TCP 拥塞控制的目的是：**防止过多的数据注入网络，避免网络中的路由器和链路发生拥塞，从而保证整体网络的稳定运行。**拥塞控制关注的是**网络整体负载情况**，而不是某一具体接收端。

**（二）实现原理**

TCP 拥塞控制通过维护一个 **拥塞窗口（cwnd）**，并结合网络反馈动态调整发送速率，核心思想是：

> **“探测可用带宽，出现拥塞就退让”**

**主要机制包括：**

1. **慢开始（Slow Start）**
   初始时 cwnd 较小，每收到一个确认报文，cwnd 指数增长，用于快速探测网络容量。
2. **拥塞避免（Congestion Avoidance）**
   当 cwnd 达到慢开始门限（ssthresh）后，cwnd 线性增长，避免过快引发拥塞。
3. **快速重传（Fast Retransmit）**
   当连续收到三个重复 ACK 时，认为发生丢包，立即重传数据，而不等待超时。
4. **快速恢复（Fast Recovery）**
   在快速重传后不重新进入慢开始，而是适当减小 cwnd，提高恢复效率。

### 三、流量控制与拥塞控制的区别

| 对比项   | 流量控制         | 拥塞控制      |
| -------- | ---------------- | ------------- |
| 控制对象 | 接收方           | 网络整体      |
| 目的     | 防止接收方被压垮 | 防止网络拥塞  |
| 依据     | 接收窗口 rwnd    | 拥塞窗口 cwnd |
| 控制范围 | 端到端           | 全局网络      |

### 四、总结

TCP 流量控制通过**接收窗口机制**协调发送方与接收方的处理能力，而 TCP 拥塞控制通过**动态调整拥塞窗口**来适应网络负载变化。两者相互配合，使 TCP 能在保证可靠性的同时，实现高效而稳定的数据传输。

## Q6：带宽

### 一、链路带宽（Link Bandwidth）

**（一）概念**

**链路带宽**是指一条网络链路在**理想情况下**所能提供的**最大数据传输速率**，通常由链路的物理特性和链路层技术决定，是一种**固定的理论上限**。

链路带宽反映的是链路的**容量上限**，与当前网络负载无关。

常用单位为：bps、Mbps、Gbps 等。

**（二）测量方法**

链路带宽通常通过以下方式获得：

1. **根据物理与链路层标准确定**
   例如：
   - Fast Ethernet：100 Mbps
   - Gigabit Ethernet：1 Gbps
2. **专用点对点测试（无竞争）**
   在链路空闲、无其他流量干扰的情况下，持续发送数据流，测得的最大稳定吞吐率可近似视为链路带宽。
3. **配置或设备参数读取**
   通过交换机、网卡或路由器的接口配置参数直接获取额定带宽。

### 二、可用带宽（Available Bandwidth）

**（一）概念**

**可用带宽**是指在某一时刻、某一路径上，**除去已有网络流量后仍可供新流量使用的带宽**。

可用带宽是一个**动态变化的量**，受当前网络负载、竞争流量和拥塞状况影响。

通常可表示为：

> 可用带宽 = 链路带宽 − 已被占用的带宽

**（二）测量方法**

可用带宽无法通过配置直接获得，一般采用**主动测量方法**进行估计：

1. **探测包对法（Packet Pair / Packet Train）**
   - 发送成对或成组的数据包
   - 根据接收端包间隔的变化推断链路剩余带宽
   - 常用于端到端路径测量
2. **发送速率自适应法**
   - 逐步增加发送速率
   - 观察是否出现排队延迟或丢包
   - 当性能明显下降时，认为已超过可用带宽
3. **基于时间戳与排队延迟分析**
   - 通过测量往返时延（RTT）的变化
   - 判断链路是否出现拥塞，从而估计可用带宽

### 三、总结

链路带宽描述的是链路的**物理或技术极限能力**，而可用带宽反映的是网络在**当前运行状态下的实际可用资源**。在实际网络性能分析中，可用带宽比链路带宽更能反映用户真实体验。

## Q7 QoS模型

### 一、综合服务模型（IntServ）与区分服务模型（DiffServ）的区别

**（一）综合服务模型（Integrated Services, IntServ）**

**核心思想：**
**每一条流（flow）提供端到端的明确 QoS 保证**。

**主要特点：**

1. 以**流**为单位进行资源管理
2. 需要在网络中**预留资源**（带宽、缓存等）
3. 路由器需要为每条流**维护状态信息**
4. 通过 RSVP 协议进行信令和资源预留

**优点：**

- QoS 保证精确
- 可提供严格的实时服务（如语音、视频）

**缺点：**

- 路由器状态开销大
- 信令复杂
- **扩展性差，不适合大规模网络**

**（二）区分服务模型（Differentiated Services, DiffServ）**

**核心思想：**
不再区分具体流，而是将业务划分为**不同服务等级（Class）**。

**主要特点：**

1. 不维护逐流状态
2. 在 IP 头部使用 **DS 字段**标记业务类型
3. 路由器根据业务类别执行不同的转发策略（PHB）
4. 核心网络只做简单快速转发

**优点：**

- 可扩展性好
- 实现简单
- 适合运营级大规模网络

**缺点：**

- 只能提供**相对 QoS**
- 无法保证单个流的严格服务质量

**（三）两种模型对比总结**

| 对比项       | 综合服务模型（IntServ） | 区分服务模型（DiffServ） |
| ------------ | ----------------------- | ------------------------ |
| QoS 粒度     | 逐流                    | 按业务类别               |
| 是否资源预留 | 是                      | 否                       |
| 路由器状态   | 维护逐流状态            | 不维护逐流状态           |
| QoS 保证     | 严格                    | 相对                     |
| 扩展性       | 差                      | 好                       |
| 典型场景     | 小规模、实时应用        | 大规模运营网络           |

### 二、RSVP 的工作原理

**（一）RSVP 的作用**

**RSVP（Resource ReSerVation Protocol）** 是 IntServ 模型中的信令协议，用于在端到端路径上**预留网络资源**，从而支持 QoS 保证。

**（二）RSVP 的基本工作流程**

RSVP 的核心思想是：**发送方描述流量特征，接收方发起资源预留请求**。

**具体过程如下：**

1. **PATH 报文（发送方向接收方）**
   - 描述发送流的流量参数（如速率、突发性）
   - 沿数据转发路径逐跳传递
   - 在路由器中建立路径状态
2. **RESV 报文（接收方向发送方）**
   - 请求为该流预留资源
   - 指定所需的 QoS 参数
   - 沿反向路径逐跳返回
3. **逐跳资源检查与预留**
   - 每个路由器检查自身资源是否满足请求
   - 若满足，则分配资源并转发 RESV
   - 若不满足，则返回错误信息
4. **资源预留成功后开始数据传输**

### 三、为什么 RSVP / 综合服务模型采用“逐跳路由”机制

**（一）根本原因：资源是逐跳存在的**

网络中的关键资源（如带宽、缓存、调度能力）是**分布在每一跳路由器上的**，不存在“端到端一次性分配”的可能。

因此：

> **QoS 是否能保证，必须由路径上的每一跳共同确认。**

**（二）逐跳机制的必要性**

1. **每一跳的资源状况不同**
   - 不同路由器的负载、带宽和缓存情况不同
   - 必须逐个检查是否满足 QoS 请求
2. **路径可能动态变化**
   - RSVP 可在路径变化时重新建立预留
   - 逐跳机制更灵活
3. **便于局部失败与回退**
   - 若某一跳无法满足请求，可立即拒绝
   - 避免无效的端到端资源占用

**（三）逐跳机制的代价**

- 路由器需要维护大量状态
- 信令开销大
- 扩展性受限

这正是 **IntServ + RSVP 难以在 Internet 核心部署**，而 DiffServ 被广泛采用的根本原因。

### 四、总结

> 综合服务模型通过 RSVP 逐流预留资源，能够提供严格的 QoS 保证，但扩展性差；区分服务模型通过业务分类和差异化转发实现相对 QoS，扩展性好。RSVP 采用逐跳机制，是因为网络资源分布在各个路由器上，必须逐跳检查和分配。

## Q8：加密算法

### 一、对称加密算法（Symmetric Encryption）

**定义：**
对称加密算法是指**加密和解密使用同一把密钥**的加密算法。通信双方在通信前需共享相同的密钥。

**常见算法：**
DES、AES、3DES 等。

### 二、非对称加密算法（Asymmetric Encryption）

**定义：**
非对称加密算法使用**一对密钥**：公钥和私钥。

- 公钥可公开
- 私钥由密钥拥有者保管

二者在数学上相关，但无法由公钥推导出私钥。

**常见算法：**
RSA、ECC 等。

### 三、对称加密与非对称加密的优劣势

**（一）对称加密算法的优劣势**

**优点：**

1. 加密和解密速度快
2. 计算开销小，适合加密大量数据

**缺点：**

1. 密钥分发困难
2. 一旦密钥泄露，通信安全性完全失效
3. 难以在大规模网络中安全管理密钥

**（二）非对称加密算法的优劣势**

**优点：**

1. 不需要事先共享密钥
2. 便于密钥管理和分发
3. 可用于身份认证和数字签名

**缺点：**

1. 计算复杂度高
2. 加解密速度慢
3. 不适合直接加密大量数据

**（三）对比总结表**

| 对比项   | 对称加密   | 非对称加密        |
| -------- | ---------- | ----------------- |
| 密钥数量 | 一把       | 一对（公钥/私钥） |
| 运算速度 | 快         | 慢                |
| 密钥分发 | 困难       | 容易              |
| 适合场景 | 大数据加密 | 密钥交换、签名    |
| 扩展性   | 较差       | 较好              |

### 四、加密过程

**（一）对称加密的加密过程**

1. 发送方与接收方共享同一密钥 K
2. 发送方使用密钥 K 对明文进行加密，得到密文
3. 密文在不安全信道上传输
4. 接收方使用相同密钥 K 对密文解密，恢复明文

**特点：**

- 加密和解密过程对称
- 安全性依赖于密钥的保密性

**（二）非对称加密的加密过程**

1. 接收方生成一对密钥（公钥 PK，私钥 SK）
2. 接收方公开公钥 PK，私钥 SK 保密
3. 发送方使用接收方的公钥 PK 对明文加密
4. 接收方使用私钥 SK 对密文解密

**特点：**

- 即使公钥泄露也不影响安全
- 保证通信内容的机密性

### 五、签名过程（数字签名）

**（一）数字签名的目的**

数字签名用于实现：

1. **身份认证**
2. **数据完整性校验**
3. **不可否认性**

**（二）签名过程（非对称加密）**

1. 发送方对原始消息计算摘要（Hash）
2. 发送方使用**自己的私钥**对摘要进行加密，生成数字签名
3. 将“消息 + 签名”一起发送给接收方

**（三）验证过程**

1. 接收方使用发送方的**公钥**解密签名，得到摘要值
2. 接收方对收到的消息重新计算摘要
3. 对比两个摘要：
   - 相同 → 签名有效
   - 不同 → 签名无效

### 六、综合说明

> 对称加密适合对大量数据进行高速加密，但存在密钥分发问题；非对称加密解决了密钥分发和身份认证问题，但计算开销较大，因此在实际系统中常将两者结合使用。

## Q9：域名服务器与查询

### 一、根域名服务器的作用

**根域名服务器（Root DNS Server）** 是 DNS 分层体系结构中的**最高层服务器**，负责管理**根区（Root Zone）**，即整个域名系统的起点。

**主要作用：**

1. **提供顶级域名服务器的信息**
   根域名服务器并不保存具体域名（如 [www.example.com）](http://www.example.xn--com) ）IP 地址，而是保存各**顶级域名（TLD）**（如 .com、.edu、.cn 等）对应的权威域名服务器地址。
2. **引导域名解析过程**
   当本地域名服务器无法解析某个域名时，首先向根域名服务器查询，根服务器返回“下一步该去问谁”，从而引导查询逐层向下进行。
3. **维持 DNS 层次结构的稳定性**
   根域名服务器是 DNS 分布式体系的关键组成部分，保证整个 Internet 域名解析体系的统一与可靠。

### 二、递归查询（Recursive Query）

**（一）原理**

**递归查询**是指：

> **查询请求发出后，接收查询的服务器必须返回最终解析结果，或返回错误信息。**

在递归查询中，客户端只发起一次请求，后续所有查询过程均由服务器代为完成。

**（二）工作过程**

1. 主机向**本地域名服务器**发出递归查询请求；
2. 若本地域名服务器没有缓存结果，则：
   - 依次向根域名服务器、顶级域名服务器、权威域名服务器发起查询；
3. 获得最终 IP 地址后，将结果返回给主机；
4. 同时将解析结果缓存，以供后续查询使用。

**（三）特点**

- 对客户端透明，使用简单
- 服务器负担较重
- 常用于**主机 → 本地域名服务器**之间

### 三、迭代查询（Iterative Query）

**（一）原理**

**迭代查询**是指：

> **被查询服务器只返回自己所知道的最优信息，而不负责继续代替查询。**

查询方需要根据返回的信息，逐步向下一级服务器继续查询。

**（二）工作过程**

1. 查询方首先向根域名服务器查询；
2. 根服务器返回相应的顶级域名服务器地址；
3. 查询方向顶级域名服务器查询；
4. 顶级域名服务器返回权威域名服务器地址；
5. 查询方向权威域名服务器查询，获得最终 IP 地址。

**（三）特点**

- 服务器负担较轻
- 查询过程由请求方逐步推进
- 常用于**DNS 服务器之间**

### 四、总结

根域名服务器在 DNS 中起到**引导和索引作用**，而非直接解析域名。递归查询强调**由服务器完成全部解析过程**，简化客户端操作；迭代查询则通过**逐级询问**实现分布式解析，降低服务器负担。二者相互配合，构成了高效、可扩展的 DNS 域名解析机制。

## Q10：区块链

### 一、区块链中的哈希链

**哈希链（Hash Chain）是指在区块链中，各个区块通过哈希值首尾相连**形成的一种链式结构。

具体而言：

- 每一个区块都包含：
  1. 当前区块的数据
  2. 上一个区块的哈希值（Previous Hash）
- 当前区块的哈希值由“区块内容 + 上一区块哈希”共同计算得到

因此：

> **后一个区块通过保存前一个区块的哈希值，将所有区块按时间顺序连接起来，形成哈希链。**

### 二、哈希链的作用

**（一）保证数据不可篡改性**

由于哈希函数具有**雪崩效应**：

- 区块中任意一比特数据发生变化
- 都会导致该区块的哈希值发生巨大变化
- 进而使其后所有区块的哈希全部失效

因此，攻击者若想篡改某一区块的数据，必须同时修改该区块及其之后的所有区块，这在实际中几乎不可行。

**（二）保证区块顺序与历史一致性**

哈希链明确记录了区块之间的前后依赖关系：

- 每个区块都“指向”其前一个区块
- 区块顺序无法随意调换

从而保证了区块链中**交易历史的时间顺序和一致性**。

**（三）为共识机制提供基础**

哈希链结构为区块链中的共识机制（如工作量证明、权益证明）提供了基础数据结构：

- 节点可通过验证哈希链是否合法
- 判断某一条链是否可信
- 从而在多条链中选择“有效链”

### 三、区块链作用举例

区块链利用**哈希链 + 分布式存储 + 共识机制**，在多个场景中发挥重要作用。

**（一）数字货币（如比特币）**

- 区块链用于记录所有交易信息
- 哈希链保证交易记录不可篡改
- 无需中心化银行即可实现可信转账

**（二）供应链管理**

- 商品从生产、运输到销售的每个环节记录在区块链中
- 哈希链保证数据真实、可追溯
- 防止数据被事后篡改

**（三）数据存证与版权保护**

- 将文件哈希值写入区块链
- 利用时间戳和哈希链证明数据“在某一时间已存在”
- 用于电子证据、版权确权等场景

### 四、总结

区块链中的哈希链通过将区块按哈希值连接起来，保证了数据的**不可篡改性、顺序一致性和历史可追溯性**。基于这一结构，区块链能够在无需中心化可信第三方的情况下，实现可信的数据记录与价值交换。

## Q11：总线和无线协议

### 一、总线和无线网络使用的协议

**（一）总线型有线网络使用的协议**

**典型协议：**

- **CSMA/CD（Carrier Sense Multiple Access with Collision Detection）**

**应用场景：**

- 传统以太网
- 共享总线型有线网络（如早期 Ethernet）

**（二）无线网络使用的协议**

**典型协议：**

- **CSMA/CA（Carrier Sense Multiple Access with Collision Avoidance）**

**应用场景：**

- 无线局域网（IEEE 802.11，Wi-Fi）

### 二、CSMA/CD 与 CSMA/CA 的基本思想

**（一）CSMA/CD（用于总线）**

CSMA/CD 的核心思想是：

> **先监听信道，再发送数据；若检测到冲突，则立即停止并重传。**

其工作过程为：

1. 发送前监听信道是否空闲
2. 若空闲则发送
3. 发送过程中持续监听
4. 一旦检测到冲突，立即停止发送并进行退避重传

**（二）CSMA/CA（用于无线）**

CSMA/CA 的核心思想是：

> **尽量避免冲突，而不是在冲突发生后再处理。**

其工作过程为：

1. 发送前监听信道
2. 若空闲，则等待一个随机退避时间
3. 退避结束后发送数据
4. 可选使用 RTS/CTS 机制减少冲突

### 三、CSMA/CD 与 CSMA/CA 的对比总结

| 对比项           | CSMA/CD    | CSMA/CA       |
| ---------------- | ---------- | ------------- |
| 使用场景         | 有线总线   | 无线网络      |
| 是否检测冲突     | 是         | 否            |
| 核心策略         | 冲突后处理 | 冲突前避免    |
| 是否支持边发边听 | 是         | 否            |
| 是否解决隐藏终端 | 否         | 是（RTS/CTS） |

### 四、总结

> 总线型有线网络由于能够实时检测冲突，采用 CSMA/CD；而无线网络由于无法可靠检测冲突且存在隐藏终端问题，因此采用以避免冲突为核心的 CSMA/CA 协议。
